# Poker Bot 集成测试设计方案 (Integration Tests)

你好，同学。本方案旨在建立一套基于**真实浏览器环境**、**无 Mock 数据**的全链路集成测试体系，用于验证 Poker Bot 在 `Replay Poker` 实战环境中的稳定性与正确性。

## 1. 测试核心目标
- 验证大厅寻桌与去重逻辑在真实 DOM 变动下的可靠性。
- 验证 `TableManager` 从 WebSocket 同步状态到执行物理点击的完整闭环。
- 验证“5 秒冷却重试”、“动作去重”等关键算法在网络延迟环境下的表现。
- 保证 Bot 在达到盈利/手数上限后能正常清理资源并切换桌台。

## 2. 测试用例设计

### 方案 A：寻桌与调度测试 (Lobby & Browser Dispatcher)
| 用例 ID | 测试名称 | 预期行为 |
| :--- | :--- | :--- |
| **TC-LB-01** | 大厅初始化与寻桌 | Bot 能成功导航至大厅，扫描并抓取至少 3 个满足条件的桌台 URL。 |
| **TC-LB-02** | 访问记录 FIFO 淘汰机制 | 连续访问 6 张桌台。确认第 1 张访问过的桌台重新出现在“建议列表”中（验证 5 条记录的淘汰上限）。 |
| **TC-LB-03** | 过滤失效后的刷新重试 | 人为清空可见桌台列表。确认 Bot 能触发“大厅重连”或“报错停止”逻辑，而非死循环。 |

### 方案 B：对局执行测试 (Gameplay Execution)
| 用例 ID | 测试名称 | 预期行为 |
| :--- | :--- | :--- |
| **TC-GP-01** | 同步与动作初次执行 | 在轮到 Bot 操作时，确认日志成功输出状态 Key，HUD 刷新一次，且点击按钮动作被触发。 |
| **TC-GP-02** | 动作重试机制 (5s Retry) | 人为挂起 WebSocket 消息或模拟点击失败。确认 Bot 在 5 秒后主动发起第二次点击。 |
| **TC-GP-03** | 街道切换与去重验证 | 从 Preflop 切换到 Flop。确认 `turn_key` 自动更新，Bot 立即识别新状态并刷新 HUD，不复用旧决策结果。 |
| **TC-GP-04** | All-In 场景识别 | 进入 All-In 或带金额按钮的桌台。确认 `PlayManager` 能匹配带数值的按钮（如 All-In 250）。 |

### 方案 C：生命周期与策略切换 (Lifecycle & Stats)
| 用例 ID | 测试名称 | 预期行为 |
| :--- | :--- | :--- |
| **TC-LC-01** | 止盈/止损自动离座 | 修改配置使止盈目标极低（如 10 筹码）。确认 Bot 在一局结束后自动关闭窗口并同步统计数据。 |
| **TC-LC-02** | 任务手数限制触发 | 设置最大手数为 2。运行两手牌后，Bot 应主动触发 `exit_requested` 并清档。 |
| **TC-LC-03** | 多策略平滑切换 | 分别以 `balanced` 和 `aggressive` 模式启动 Bot。确认控制台输出对应策略的前缀（如 `[AGGRESSIVE PREFLOP]`）。 |

## 3. 技术实施细节
- **测试环境**：使用 Playwright 启动真实 Chrome 浏览器。
- **配置驱动**：每次测试前动态生成 `config/settings.yaml` 副本，注入测试参数。
- **断言方式**：
  - **日志断言**：通过 `caplog` 或重定向 `bot_logger` 捕获特定的关键日志标记（如 `[EXECUTE]`、`FIFO pop`）。
  - **DOM 断言**：检查按钮是否消失、URL 是否变更。
  - **统计断言**：通过检查 `BrowserManager._accumulated_stats` 校验数据一致性。

## 4. 运行方案
建议在 `tests/integration/` 下创建 `test_bot_full_flow.py`，采用 `pytest` 运行。
测试前需确保本地已配置 `REPLAY_POKER_EMAIL` 等环境参数，或使用已登录的 Singleton 路径。
